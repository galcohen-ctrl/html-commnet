AI-Generated Games — Product Requirements (V1 / MVP)

Branded loyalty mini-games from a merchant URL · powered by an AI scrape → plan → skin pipeline · rewards via Campaign Center

Spike: CE-3975 Owner: Gal Cohen Sprint 71 discovery Status: Draft for review Updated 2026-06-30

What we're building

Let a merchant turn their own brand into playable, branded HTML5 loyalty mini-games — with no design work and no per-merchant development. The merchant enters a website URL (or uploads assets), an AI pipeline scrapes the brand and skins games from a verified template bank, and the merchant configures and activates them. Members play in the Como app and Customer Portal and win rewards. Eligibility and rewards run on the existing Campaign Center — the same model Spin the Wheel uses today.

Merchant one-liner: Type in your website. Get branded games your members play to win your rewards — in minutes.

Why now — customer evidence

Pancake Parlour · AU · ~60 stores

Spin to Win is their best campaign ever:

  • 10.5% YoY revenue lift during campaign
  • 15–16k spins per week
  • 28–30% spin → in-store redemption

Wants profitability controls: retry limits, tier/behaviour gating.

Java Cafe · SA · ~32 stores

"We need our customer to be obsessed with the Como application… let me find a game."

  • Spend-gated games ($3+ unlocks)
  • Values AI curation over manual selection
  • Flagged the per-game config screen as unclear

Goals & non-goals

V1 Goals
  • Generate branded games from a URL — no design/dev work
  • Configurable + phone-previewable before going live
  • Members discover, play, win, redeem in app + portal
  • Reuse Campaign Center for eligibility + rewards
  • Brand artifact built generically for future agents
Non-goals (V1)
  • From-scratch dynamic game-code generation (lab only)
  • "Describe a game and AI builds it" (V2)
  • Always-on mode (triggered/tag-based only)
  • Analytics dashboards · kiosk channel · multiplayer
  • Auto re-skin on rebrand · merchant-uploaded templates

Locked V1 decisions

Confirmed with PM on 2026-06-30. These are the constraints the build inherits.

#Decision
D1Entry point: new "Games" item in the Campaign Center left sidebar. (Future note: consider moving Spin the Wheel config here too — not V1.)
D2Input: primary = merchant URL; secondary/fallback = bulk upload (drag-drop / paste / ZIP). URL is the dominant CTA.
D3Consent: legal modal before scraping; record written to the existing Hub audit table.
D4Scrape progress: progressive asset gallery, not a plain spinner.
D5Partial scrape: ask merchant to fill gaps instead of failing.
D6Selection model: Hybrid — planner picks 3 games per batch; merchant approves / regenerates / requests more. Bank size hidden.
D7Template bank: 10 verified HTML5 templates in V1.
D8Per-game config: win score, wins-allowed, reward (Benefit Picker), eligibility (Filter Members). No difficulty field. Phone preview on the right.
D9Activation: per-row Activate button (status/tag-based). No global "Publish" button.
D10Eligibility ownership: Games UI owns it. Activation auto-creates 1 eligibility + 1 reward rule in CC, shown read-only with a "Managed by Games" badge.
D11One rule set per game. Disabling a game disables its CC rules.
D12Member discovery (V1): home card (only with a game_*_eligible tag) + push/deep-link. Games tab = V2. Never replace Spin the Wheel.
D13Single vs Arena: 1 eligible game → launch directly. 2+ → minimal Games Arena chooser.
D14Play/win: unlimited plays until a win; wins-allowed caps reward wins; untag on final win. MVP fallback: hard-code 1 win.
D15Lifecycle: scrape once (first run); later batches reuse assets + add table rows; disabled games keep assets (re-activatable free).
D16Max-out: at 10 templates used, Generate button disabled → replaced with fake-door feedback CTA.
D17Mode: triggered/tag-based only (Spin-the-Wheel-consistent). "Always-on" simulated via a low-bar eligibility rule.

Architecture & integration

Components

  • Hub 2 Games UI — empty state, consent, gallery, gap-fill, config, table
  • Brand Scraping Service — URL + upload → canonical artifact
  • Brand Understanding Store — reusable schema for future agents
  • Agent pipeline — Planner + Coder (parallel), reuse ADK shape
  • Template Bank — 10 verified templates + contract
  • Game Hosting Service — serve, token-auth, event bridge, untag
  • Campaign Center integration — auto rules + GAME_WIN
  • Member touchpoints — Phoenix webview + CP iframe

postMessage contract

Game → host:

GAME_READY GAME_STARTED SCORE_UPDATE LEVEL_PASSED GAME_OVER {won,score}

Host → game:

INIT {config} PAUSE RESUME

On GAME_OVER {won:true} the host calls the existing submitEvent API → CC reward rule fires.

Data model (sketch)

  • brand_understanding — merchantId, sourceType, designMd, tokensJson, assetManifest, refreshedAt
  • game — gameId, merchantId, templateId, name, status, winScore, winsAllowed, rewardBenefitId, eligibilityFilterId, assetPath
  • game_win — gameId, memberId, winCount, lastWonAt or reuse CC event store — Q3
  • CC: eligibility_rule, reward_rule tagged "Managed by Games"
